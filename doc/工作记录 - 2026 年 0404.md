## 刷新页面后的 stream 续流恢复

### 改动背景

当前页面在流式返回中途刷新，会直接丢掉原来的 SSE 连接。根因不在消息渲染，而在链路模型本身：

1. 前端之前只靠 `SharedWorker` 托管进行中的流；
2. 一旦页面刷新，`SharedWorker` 可能跟着被销毁；
3. 后端原来的 `POST /api/v1/chat/sessions/{session_id}/stream` 绑定的是单条 HTTP 连接，连接断开后生成也会一起停掉；
4. 所以“刷新后继续接上刷新前那条流”在旧实现里并不成立。

这轮参考 deer-flow 的思路，把能力补成“后端托管活动流 + 前端按 cursor 续上”，而不是继续依赖纯前端保活。

### 主要修改点

前端修改文件：

- `src/services/chatService.ts`
- `src/services/chatStreamBridgeService.ts`
- `src/services/chatStreamSnapshotStore.ts`
- `src/workers/chatStreamWorker.ts`
- `src/pages/Chat/ChatPage.tsx`
- `src/pages/Partner/PartnerPage.tsx`
- `tests/chatService.test.ts`
- `tests/chatStreamReconnect.test.ts`

后端修改文件：

- `/Users/yang/guoren-ai/app/api/chat.py`
- `/Users/yang/guoren-ai/app/services/chat_stream_runtime.py`

主要内容：

- 前端 `readSseStream()` 新增 `id:` 解析能力，能拿到每条 SSE 事件的 cursor
- 前端新增活动流快照存储，按 `sessionId` 持久化 `messages + activeMessageId + lastEventSequence`
- 页面刷新后，先恢复本地快照，再尝试：
  1. 订阅现存 `SharedWorker` 快照
  2. 若 worker 已丢失，则走后端 `resume-stream` 继续接收后续事件
- `SharedWorker` 新增 `resume-stream` 指令，支持按 cursor 继续消费后端流
- 停止按钮除了中断本地 reader，也会调用后端 stop 接口，避免后台任务继续跑
- 后端新增会话级流式运行时管理器，把单次流式任务放到后台 task 执行
- 后端 `POST /sessions/{session_id}/stream` 现在负责“启动任务 + 订阅事件”
- 后端新增 `GET /sessions/{session_id}/stream?cursor=...`，用于刷新后按 cursor 续流
- 后端新增 `POST /sessions/{session_id}/stream/stop`，用于主动停止后台流式任务
- 后端会为每条 SSE 事件补 `id: <sequence>`，前端据此续流，不会重复消费旧事件

### 异常处理与兼容处理

- 仍然保留原有 `SharedWorker` 路径，同页面内的恢复不受影响
- 如果刷新时 worker 还活着，优先直接从 worker 快照恢复，避免额外后端重连

## 刷新续流的 Redis 补强

### 改动背景

上一轮已经把“刷新后继续接收 stream”从纯前端保活，升级成了“后端托管活动流 + 前端 cursor 续流”。

但那版后端运行时还是单进程内存模型：

1. 当前实例生成事件并缓存到本地内存；
2. 刷新后如果请求刚好还落到同一个实例，可以续上；
3. 如果刷新后落到别的实例，另一个实例拿不到原实例内存里的事件日志，续流能力还是会退化。

这轮继续补的是“跨实例恢复”这一层，不改前端接口，只把后端活动流事件日志下沉到 Redis。

### 主要修改点

修改文件：

- `/Users/yang/guoren-ai/app/services/chat_stream_runtime.py`
- `/Users/yang/guoren-ai/app/api/chat.py`
- `/Users/yang/guoren-ai/tests/test_chat_stream_runtime_redis.py`

主要内容：

- `chat_stream_runtime` 新增 Redis 依赖，使用异步 Redis 客户端
- 新增三类 Redis key：
  - `chat_stream:events:{session_id}`：SSE 事件日志
  - `chat_stream:state:{session_id}`：活动流状态
  - `chat_stream:stop:{session_id}`：跨实例 stop 标记
- 每次 SSE 事件发布时，同时写入 Redis 列表
- `stream_events()` 除了本地内存订阅，也支持从 Redis backlog 读取并轮询后续增量
- `has_runtime()` 不再只看本地内存，也会看 Redis 状态
- `stop()` 不再只停本地 task，也会写 Redis stop 标记
- `forward_stream_chunks_to_runtime()` 在转发过程中会检查 stop 标记，收到后停止后续事件转发

### 异常处理与兼容处理

- 同实例内仍优先走本地内存订阅，减少 Redis 轮询开销
- 跨实例恢复时退化成 Redis backlog + 轮询 tail，不依赖 sticky session
- Redis key 都带 TTL，避免活动流完成后永久堆积
- 当前 stop 对跨实例是“下一批 chunk 前生效”的模型，不是强杀底层 LLM 连接
- 这轮没有改前端协议，前端无需额外适配

### 涉及模块

- 后端活动流运行时
- 后端 stream 转发逻辑
- Redis 活动流事件存储

### 可能影响范围

- 多实例部署下的刷新续流稳定性
- 跨实例 stop 的生效时机
- Redis 读写压力会比上一轮略高

## 首轮自动发消息重复建会话修复

### 改动背景

用户从首页直接发起首轮对话时，页面会先跳到 `/chat`，然后由 Chat 页的 `useEffect` 自动发送首条消息。  
当前项目入口开着 `React.StrictMode`，开发态会故意把 mount effect 跑两次。  
另外，这个首轮 effect 还依赖 `routeSessionId`，一旦前端在请求中途把新 `sessionId` 同步回地址栏，effect cleanup 也会跟着触发。

这就导致首轮链路变成：

1. 第一次 effect 执行，调用 `createChatSession`，先落一条空会话；
2. cleanup 里直接 `abortController.abort()`；
3. `StrictMode` 重挂载或 `routeSessionId` 写回地址栏时，这个 cleanup 会把首轮请求提前掐掉；
4. 如果再让第二次 effect 继续执行，就会变成“多一条空会话”；如果把第二次 effect 硬拦掉，就会变成“没有重复记录，但 stream 不通”。

这个问题主要出现在“首页首轮自动发送”这条链路，手动点新建后再发消息不走这段 effect，所以表面上看起来是“只有 0 会话时首轮会多一条空记录”。

### 主要修改点

修改文件：

- `src/pages/Chat/ChatPage.tsx`
- `src/pages/Partner/PartnerPage.tsx`
- `tests/chatInitialPromptDedup.test.ts`

主要内容：

- Chat / Partner 页的首轮自动发送不再直接在 effect 里同步执行
- 改成 `window.setTimeout(..., 0)` 延后到下一个 tick 再发起
- effect cleanup 只做 `window.clearTimeout(...)`
- 这样 `StrictMode` 第一轮重挂载只会取消第一次调度，不会把真正已经开始的请求 abort 掉
- 也避免了 `sessionId` 写回地址栏时，effect cleanup 顺手把流式请求掐掉

### 异常处理与兼容处理

- 只调整首页首条“自动发送” effect 的触发时机，不改用户手动点击发送的链路
- 只在前端修正副作用时机，不改现有接口协议，也不改后端 session 落库逻辑
- 继续保留 `location.state -> /chat` 这条跳转方式，避免扩大改动范围

### 涉及模块

- Chat 页首页首轮自动发送
- Partner 页首页首轮自动发送
- 首轮自动发送去重与避免误 abort 的静态测试

### 可能影响范围

- 首页输入框直接发起的首轮对话
- 伙伴页如果后续复用同样的首页跳转自动发送链路，也不会再被 StrictMode 双触发
- 不影响已有 session 内的继续追问和手动发送
