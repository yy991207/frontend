## Chat Think 实时渲染补漏

### 改动背景

后端新的 SSE 已经开始稳定返回 `reasoning_content`，但前端实际页面仍然看不到 think。排查后确认不是消息组件没能力渲染，而是页面默认优先走 `SharedWorker` 这条共享 stream 通道，这条链之前没有把 `reasoningContent` 带进快照。

结果就是：

- 直连 `readSseStream` 的分支能收到 think
- 真正运行时常走的 `streamBridge / chatStreamWorker` 分支把 think 丢了
- 所以 process 区块看起来像“没有渲染 think”

### 主要修改点

修改文件：

- `src/services/chatStreamBridgeService.ts`
- `src/workers/chatStreamWorker.ts`
- `tests/chatReasoningIntegration.test.ts`

主要内容：

- 给 `StreamBridgeMessage` 补充 `reasoningContent?: string | null`
- SharedWorker 广播快照时保留 `reasoningContent`
- 流式接收 `onReasoningDelta` 时，把 think 片段累加到当前 assistant 消息
- 新的 followup assistant 消息显式初始化 `reasoningContent: null`
- 增加共享 stream 通道的测试，防止以后再次出现“后端有 think，页面没渲染”的回归

### 异常处理与兼容处理

- 这次只补共享 stream 通道，不改现有消息分组和 process 组件结构
- 如果后端没有返回 `reasoning_content`，SharedWorker 仍按原来的正文流工作
- 历史记录接口原本就已经映射 `reasoning_content -> reasoningContent`，这次不额外改动历史页逻辑

### 涉及模块

## Chat Stream 第二轮 Think 顺序修正

### 改动背景

当前后端 SSE 的真实顺序已经是：

- `on_chat_model_start`
- think
- tool
- 第二轮 `on_chat_model_start`
- tool 后 think
- 正文

但前端实时流式这条链路里，只有正文 token 到来时才会切新的 assistant 消息，导致“tool 后再次出现的 think”先被追加进旧消息，等正文到了才另起一条消息，所以页面实时顺序会乱。  
历史记录正常，是因为后端最终落库后的消息边界本来就是对的。

## Chat Process 单框合并

### 改动背景

虽然上一轮已经把实时流式的消息顺序修正了，但页面展示层仍然会把一轮回复里的多段 processing 拆成多个框：

- 第一段 think + tool 一个框
- tool 后第二段 think 再单独一个框
- 最后正文单独一个框

这和 deer-flow 的展示方式不一致。  
deer-flow 的做法不是“每条 processing message 一个框”，而是把连续的 processing message 合并成一个 `assistant:processing` 组，再统一渲染成一个 process 面板。

### 主要修改点

修改文件：

- `src/core/messages/utils.ts`
- `src/components/chat/message-group.tsx`
- `tests/chatMessageGrouping.test.ts`
- `tests/chatDeerflowRendering.test.ts`

主要内容：

- 调整 `groupMessages()`：
  - 如果上一组已经是 `assistant:processing`
  - 当前消息也还是 processing
  - 就继续合并进上一组，而不是新开一个 processing 组
- 调整 `ProcessingMessage`：
  - 从只接收单条 `message`
  - 改成接收整组 `messages`
  - 再按消息顺序统一展开 reasoning / tool / courses 等步骤
- `MessageGroupSection` 的 `assistant:processing` 分支改为整组只渲染一个 `ProcessingMessage`

### 异常处理与兼容处理

- 这次没有改流式消息切分逻辑，只改展示分组
- assistant 正文仍然保持单独的 `assistant` 组，避免把最终正文塞进 process 面板
- 为了避免相同 message id 导致步骤 key 冲突，process step 的 key 增加了消息序号后缀

### 涉及模块

- `src/core/messages/utils.ts`
- `src/components/chat/message-group.tsx`
- `tests/chatMessageGrouping.test.ts`
- `tests/chatDeerflowRendering.test.ts`

- Chat 页 processing 面板展示方式

## 会话列表实时刷新 session_name

### 改动背景

当前新会话在首轮消息发送后，左侧历史列表虽然能立刻出现新会话，但显示名称还是默认的“话题”。  
只有手动刷新页面后，前端重新调用 `/api/v1/chat/sessions`，才能拿到后端已经写回数据库的 `session_name`。

这轮目标不是改列表渲染，而是在消息回复完成后，主动触发一次左侧历史列表静默刷新，让最新 `session_name` 自动显示出来。

### 主要修改点

修改文件：

- `src/components/ChatSessionHistory/ChatSessionHistory.tsx`
- `src/pages/Chat/ChatPage.tsx`
- `src/pages/Partner/PartnerPage.tsx`
- `tests/chatSessionRendering.test.ts`

新增文件：

- `src/services/chatSessionEvents.ts`

主要内容：

- 新增 `chat-session-history-refresh` 事件和 `notifyChatSessionHistoryRefresh()`
- `ChatSessionHistory` 监听这个事件，收到后用 `loadSessions({ silent: true })` 静默重拉会话列表
- Chat 页在当前回复完成后主动通知左侧历史列表刷新
- Partner 页也同步接入同样的刷新通知
- SharedWorker 流式分支在收到 `completed` 快照时，同样触发一次刷新，保证刷新页面后恢复中的会话也能同步更新名称

### 异常处理与兼容处理

- 静默刷新不会把左侧历史面板切回“加载中”，避免用户正在使用时出现闪烁
- 只在回复完成时通知刷新，不在每个 token 到来时重复请求列表接口
- 后端本次不需要改动，因为 `/api/v1/chat/sessions` 已经会返回数据库里最新的 `session_name`
