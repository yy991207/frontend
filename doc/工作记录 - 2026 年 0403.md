## 外部 skill_output 的 JSON 文件按课程表渲染

### 改动背景

当前 `skill_output` 里的外部文件卡片点击后，会统一进入右侧 `ArtifactFileDetail` 预览面板。  
HTML 和 Markdown 已经有专门的预览逻辑，但外部 `course_table.json` 还是按普通代码文件展示，所以右侧只会看到原始 JSON 文本，不符合“课程表结果区”的展示预期。

这次目标是把外部 JSON 里的课程表结构直接渲染成可读的课程表视图，同时保留原始代码查看能力。

### 主要修改点

#### 1. 新增课程表 JSON 解析器

新增文件：

- `src/core/artifacts/course-table.ts`

主要内容：

- 新增 `parseCourseTableArtifact`
- 只识别 `type: "course_table"` 的 JSON 结构
- 校验 `query`、`total_duration`、`courses`、`resource_id`、`title`、`duration` 等关键字段
- 解析失败时返回 `null`，不影响普通 JSON 文件走原来的代码模式

#### 2. ArtifactFileDetail 新增结构化课程表预览

修改文件：

- `src/components/chat/artifact-file-detail.tsx`

主要内容：

- 增加 `CourseTablePreview`
- 当文件类型是 `json` 且内容能被解析为课程表时，默认切到 `preview` 视图
- 顶部展示课程主题、课程数量、总时长
- 列表展示每门课程的标题、时长和资源 ID
- 保留“代码 / 预览”切换，方便继续查看原始 JSON
- “新窗口”按钮命中课程表预览时，不再打开原始 JSON，而是整页打开渲染后的课表视图，方便全屏查看

#### 3. 外部 JSON 继续走 preview API 获取源码

修改文件：

- `src/components/chat/artifact-file-detail.tsx`

主要内容：

- 外部 HTML / Markdown / JSON 统一沿用 preview API
- 外部 `json` 通过 `/files/preview?url=...` 代理拿到源码文本
- 这样既能绕开 OSS 的 CORS 限制，也能继续在前端本地把课程表 JSON 解析成结构化视图

#### 4. 新增课程表预览样式

修改文件：

- `src/components/chat/artifacts.module.less`

主要内容：

- 新增 `artifactStructuredWrap`
- 新增课程表头图区、统计卡片、课程列表卡片样式
- 视觉继续沿用蓝青方向，避免紫色渐变
- 补充移动端布局适配

### 异常处理与兼容处理

- 只有命中 `course_table` 结构时才切换成课程表视图，其他 JSON 仍然按代码文件处理
- 外部 JSON 读取失败时，仍然走原来的“文件内容加载失败”提示，不影响主聊天区
- 历史消息和流式消息不需要单独改链路，因为它们最终都会走 `skill_output -> onOpenFile -> ArtifactFileDetail`
- 针对后端 preview API 偶发返回的“非严格 JSON”文本（如标题里包含未转义双引号），前端课程表解析器增加了固定结构兜底解析，避免右侧继续退回源码视图

### 改动背景

用户继续反馈 deerflow 风格里最关键的不是“有个统一工具卡”，而是不同工具的结果项本身长得不一样。

上一版虽然已经有统一的 processing step 外壳，但 `web_search`、`explore`、`read_file` 仍然共用一套通用“结果项 + 详情”模板，看起来还是偏 frontend 自己的通用卡，不像 deerflow 那种按工具类型切开的结果区。

### 主要修改点

#### 1. 工具结果改成按类型分发

修改文件：

- `src/components/chat/tool-call.tsx`

主要内容：

- 新增 `renderWebSearchResult()`
- 新增 `renderExploreResult()`
- 新增 `renderReadFileResult()`
- 新增按 `toolCall.name` 分发的 `renderToolSpecificPreview()` 和 `renderToolSpecificDetails()`

现在会优先走：

- `web_search` -> 搜索结果卡
- `explore` -> 推荐结果卡
- `read_file` -> 文件路径 + 内容摘要卡

如果命中不了，再继续走原来的通用预览和通用详情兜底。

#### 2. 给不同结果卡补各自字段归一

修改文件：

- `src/components/chat/tool-call.tsx`

- `explore` 会提取 `title / description / duration / resource_id`

这样做的目的，是尽量兼容后端不同工具输出形态，不把结果渲染死绑在单一字段结构上。

#### 3. 专门补搜索卡、推荐卡、文件卡样式

## assistant 正文与 toolcall 顺序修正

### 改动背景

用户继续反馈当前前端渲染后的顺序和实际 stream / history 返回顺序不一致。

典型现象是：

- 实际流式里先返回 assistant 正文，再触发 `rag_list_videos`
- 历史记录里同样是 `content` 在前，`tool_calls` 在后
- 但前端因为把 processing 区固定放在正文上面，所以最终视觉上总是 toolcall 先显示

进一步排查后发现根因不止一层：

1. `message-group.tsx` 里 `ProcessingMessage` 把 processing 固定渲染在正文上面；
2. `groupMessages()` 还会把同一条 assistant 里“正文 + processing”压成只剩 processing 优先；
3. 流式过程中，前端一直把多轮 `on_chat_model_stream` 的正文都追加到同一个 loading message 里，工具前后的正文会被揉成一条消息，时间顺序进一步丢失。

### 主要修改点

#### 1. groupMessages 改成正文先、processing 后

修改文件：

- `src/core/messages/utils.ts`

主要内容：

- 去掉之前对连续 assistant 消息的强制合并
- 新增 `hasAssistantOutput()`
- 当同一条 assistant 同时有正文和 processing 时，先产出 `assistant` 组，再产出 `assistant:processing` 组
- 只有纯 loading 占位时，才继续走 `assistant:loading`

这样像：

- 文字
- toolcall
- 后续文字

这类顺序至少在分组层不会再被直接打乱。

#### 2. ProcessingMessage 不再顺手把正文也渲进去

修改文件：

- `src/components/chat/message-group.tsx`

主要内容：

- `ProcessingMessage` 现在只负责 deer-flow 的步骤流
- assistant 正文、技能文件卡和复制按钮只留在普通 assistant 分支里渲染

这样分组层既然已经把“正文”和“processing”拆成前后两个区块，组件层就不能再把正文塞回 processing 下面，不然顺序还是会错。

#### 3. 新增流式正文续写 helper，工具后正文自动切到新消息

新增文件：

- `src/core/messages/streaming.ts`

主要内容：

- 新增 `appendTextDeltaToStreamMessages()`
- 当当前 assistant 消息已经进入工具步骤后，后续新的正文 chunk 不再继续追加到原消息
- 改成自动创建 follow-up assistant message

这一步是这轮最关键的修复。因为真实链路里经常是：

- 第一段正文
- tool start / tool end
- 第二段正文

如果还一直往同一个 message 里拼文本，哪怕 UI 分组修了，顺序还是会错。

```
skill_output: [{ url: "https://oss.../hello.html", filename: "hello.html" }]
  ↓
SkillOutputCard 点击 → onOpenFile(item.url, item.url)
  ↓
handleOpenFile(filepath, originalUrl) → { filepath, sessionId, baseUrl, originalUrl }
  ↓
ArtifactFileDetail 检测到 isExternalUrl = true
  ↓
loadPreviewContent({ baseUrl, sessionId, url: originalUrl })
  → GET /api/v1/chat/sessions/{sessionId}/files/preview?session_id=xxx&url=https://oss...
  ↓
返回 HTML 源码 → <iframe srcDoc={content}> 渲染
```

### 异常处理和兜底方式

- 如果 preview API 请求失败，显示"文件内容加载失败"错误提示
- 非代码类型的外部 URL（如图片、PDF）仍然走 iframe 直接加载外部 URL
- `originalUrl` 缺失时不触发 preview 链路，不影响现有行为

## Markdown 中 OSS URL 链接卡片化渲染 + 课表类型识别

### 改动背景

Markdown 文本中出现的 OSS URL（如 `https://guoren-skills-test.oss-cn-beijing.aliyuncs.com/agent_results/course_table_education.json`）默认被 Streamdown 渲染为普通蓝色文字链接，视觉效果差。需要：
1. 识别 OSS URL 并渲染为文件卡片样式
2. 针对课表 JSON（`course_table_*.json`）特殊识别，显示"课表"标签 + 绿色图标

### 主要修改点

#### 1. MarkdownContent 自定义链接渲染器

修改文件：

- `src/components/chat/markdown-content.tsx`
- `src/components/chat/markdown-content.module.less`（新增）

主要内容：

- 通过 Streamdown `components` prop 传入自定义 `LinkRenderer`
- `isOssUrl(href)`：检测是否为 OSS URL
- `isCourseTableUrl(href)`：检测文件名是否包含 `course_table` 且以 `.json` 结尾
- 课表链接：绿色渐变图标 + `UnorderedListOutlined` + "课表" badge
- 普通文件链接：蓝色渐变图标 + `FileTextOutlined` + 扩展名类型

### 类型判断逻辑

```
URL → 提取文件名 → 判断:
  - filename.includes('course_table') && filename.endsWith('.json') → 课表卡片
  - isOssUrl(href) → 普通文件卡片
  - 其他 → 默认 <a> 链接
```

### 涉及模块

- `src/components/chat/markdown-content.tsx`
- `src/components/chat/markdown-content.module.less`

### 可能影响范围

- 所有 Markdown 渲染场景（聊天消息、预览面板、处理中消息等）中的 OSS URL 链接样式
- 课表 JSON 链接显示为绿色课表卡片
